# chat-harness — Design Doc

**Status:** Draft design doc, opened 2026-07-09. Not built yet. Import into WSL to build.

**One-sentence pitch:** A Windows-launched, WSL-hosted Python harness that talks to local Qwen over Ollama, spawns the same MCP servers Claude Code uses, and reads the vault's `skills/*/SKILL.md` files directly — replicating Claude Code's chat/research workflow on a fully private stack.

---

## Motivation

The two AI workflows split cleanly by intent:

| Workflow | Primary use | Tool |
|---|---|---|
| **Coding** | Repo-scoped edits, ralph loops, TDD, one-shot code | Aider + qwen2.5-coder → [[agent-harness — Design Doc]] |
| **Chat / vault** | Ask questions of the vault, ingest sources, run skills like `/recall` `/brief` `/todo` `/autoresearch` | Claude Code today → chat-harness (this doc) tomorrow |

The coding side already has a viable OSS answer (aider + ralph, both mature). The chat side does not. Aider has no MCP client, Ollama has no MCP client, and none of the ready-made local IDE integrations (Continue.dev, Cline) match Claude Code's vault-integrated skill + slash-command + sub-agent model.

This is the gap chat-harness fills.

**Only build this if the goal is a private, off-internet replica of Claude Code's vault workflow.** For anything less ambitious, `Local LLM Setup for Coding.md` already covers the ~70% that's reachable with aider + shell glue.

---

## Design principle

**Faithful mirror of Claude Code's shape, run locally.**

- Same MCP servers, spawned the same way (stdio, per MCP spec)
- Same skills, read from the same vault path
- Same slash-command convention (`.claude/commands/*.md` → `/foo`)
- Same TaskCreate pattern (TODO-list-as-tool)
- Same sub-agent pattern (parallel scoped agents)

The only substitutions: Anthropic API → Ollama, Claude → Qwen2.5 14B Instruct.

The bet: if the interface is identical, muscle memory carries over, and any skill authored for Claude has a chance of running here. When Qwen isn't strong enough, the skill fails visibly rather than silently — you fall back to Claude for that one, iterate on the prompt, or mark the skill `qwen: skip`.

## Layer ownership matrix

| Concern | Owner | Why |
|---|---|---|
| Model inference | Windows Ollama | GPU-native, no WSL CUDA shim; same VRAM pool as any Windows GPU app |
| Agent loop | `chat_harness/agent.py` (WSL Python) | Where tool-call parsing + dispatch lives |

| Slash-command parsing | `chat_harness/skills.py` reads vault `.claude/commands/*.md` | Matches Claude Code's registry pattern |
| Session state | REPL loop holds conversation in memory; one-shot is stateless | Claude Code has both; harness matches |
| Task tracking | `chat_harness/tasks.py` reads/writes `TODO.md` in CWD | Trivial TaskCreate equivalent |
| Sub-agent orchestration | `chat_harness/subagent.py` — `asyncio.gather` on scoped agent instances | Ollama handles concurrent requests |

| Config | `mcp.json` + `config.yaml` at repo root | mcp.json mirrors Claude's; config.yaml adds harness-specific keys |

## Key design decisions

### 1. Vault is the single source of truth for skills

`skills/*/SKILL.md` and `.claude/commands/*.md` live in the Obsidian vault. The harness reads them at startup — never copies, never forks.

Consequence: a skill improved during a Claude session is immediately available to the harness on next run, and vice versa. When a skill only works on Claude (because Qwen can't follow its 20-step prose reliably), add a frontmatter flag:

```yaml
qwen: skip     # or qwen: degraded — attempt but expect breakage
```

The harness reads this and either hides the slash command or shows it with a warning banner. Do not fork the skill file. Fix the skill in one place.

### 2. MCP servers spawned stdio-style, config mirrors `.mcp.json`

The harness's `mcp.json` looks structurally identical to Claude Code's `.mcp.json` — same `command`, `args`, `env` shape. The Python `mcp` SDK handles the stdio transport, tool discovery (`list_tools`), and tool invocation.

Tool schemas are discovered at boot, not hardcoded. Add a new MCP server to `mcp.json` and it's usable next session — same story as Claude.

### 3. Qwen function-calling via OpenAI-compat endpoint

Ollama exposes `localhost:11434/v1/chat/completions` with the OpenAI-compatible schema. Qwen2.5-Instruct is trained on function-call formatting (`<tool_call>{...}</tool_call>`).

The loop:
1. Send system + user prompt + tool schemas (JSON schema, OpenAI style)
2. Read response, parse `tool_calls` array (or the model's native `<tool_call>` tokens if it doesn't fill the standard field)
3. Dispatch each call via `mcp_client.py` or a local tool (`tasks.py`, etc.)
4. Feed results back as `tool` messages
5. Loop until model responds with no tool calls

**The fragile part** is step 2 — Qwen's JSON output is inconsistent under long context. Mitigation: strict JSON schema validation + retry-on-parse-error with a corrective message ("your last tool call did not parse, emit exactly one tool_call in this shape: ...").

### 4. WSL runtime, Windows launcher

The Python process runs in WSL because:
- Node MCP servers install cleanly with `apt` + `npm`, no Windows npm gotchas
- Playwright's Chromium install is scripted for Linux
- The `mcp` Python SDK is Linux-first

But it's *launched* from Windows because that's where the user is (Windows Terminal, VS Code, browser). A Windows `.bat` or a Windows Terminal profile runs:

```
wsl -e bash -c "cd ~/chat-harness && python -m chat_harness $@"
```

Ollama on Windows is reached from WSL over `localhost:11434` (mirrored networking, per `Local LLM Setup for Coding.md:117-132`). The vault is reached from WSL via `/mnt/c/Users/I538340/OneDrive - SAP SE/Documents/Obsidian Vault/`.

### 5. Both REPL and one-shot UX

Two entry modes on the same binary, matching Claude Code:

- `chat-harness` — interactive REPL. Prompt, response, prompt, response. Slash commands work. `Ctrl+D` exits.
- `chat-harness --prompt "..."` — one-shot. Stateless. Prints response, exits. For scripting.

Session state (in REPL) is just an in-memory `list[Message]`. No persistence yet — v1 is single-session. Add a `--resume` flag later if it turns out to matter.

### 6. Sub-agents via asyncio, not subprocess

`subagent.py` spawns N `agent.run(prompt, tools_subset)` coroutines with `asyncio.gather`. Each has its own conversation state but shares the MCP client pool (MCP servers are long-lived subprocesses at the harness level).

Trade-off: no process isolation. If Qwen goes wild in a sub-agent, it can hammer the same MCP servers as the parent. Acceptable for v1 — matches Claude Code's Agent tool behavior, and sub-agent misbehavior is bounded by tool-level permissions (obsidian-vault MCP won't let anyone `rm -rf` the vault; playwright MCP is just a browser).

## Skills strategy

**V1 target: full parity attempt.** Every skill in `<vault>/skills/*/SKILL.md` is loaded and registered as a slash command via its matching `.claude/commands/<name>.md` entry.

Realistic expectation, called out here so v1 isn't judged against Claude:

| Skill class | Turns | Qwen viability |
|---|---|---|
| Read-only vault queries (`/recall`, `/hot`, `/todo`, `/brief`, `/think`) | 2–8 tool calls | **Works well.** Simple retrieval + summarization. |
| Append-only writes (`/log`, `/save`, `/eod`) | 5–10 tool calls | **Works, needs prompt care.** Qwen occasionally over-writes when asked to append; strict prompt discipline needed. |
| Multi-step ingest (`/wiki-ingest`, `/delta-ingest`, `/entity`) | 15–30 tool calls with branching | **Fragile.** Expect breakage on complex sources. Iterate on prompts per-skill. |
| Multi-artifact generation (`/pptx`, `/autoresearch`, `/canvas`) | 20–50+ tool calls | **Likely broken v1.** These push Qwen past its reliable tool-loop length. |
| External-tool-heavy (`/verify-help-portal`, `/sharepoint-search`, `/customer-qa`) | Depends on Playwright reliability | **Mixed.** Playwright works; Qwen's ability to reason about SAP portal DOM is limited. |

The `qwen: skip` frontmatter flag is the escape hatch for skills that never work. Better to hide them than to run and fail silently.

## Load-bearing risks

### 1. Qwen tool-call reliability degrades past ~10 turns

Well-known Qwen2.5-Instruct behavior. Mitigations:

- Strict JSON schema validation on every tool call; retry once with a corrective message before failing
- Aggressive context trimming — the harness rewrites the message history each turn to keep only (a) system prompt, (b) last user message, (c) last 3 tool exchanges. Full history stored to disk for debugging, not fed back to Qwen.
- Per-skill turn budgets — a skill declares in its frontmatter `max_turns: 15`; harness aborts and reports past that.

### 2. Playwright in WSL is heavier than on Windows

The Playwright MCP server's Chromium install in WSL is ~500 MB and needs `libnss3`, `libatk`, etc. First-time install is slow and can fail on Ubuntu variants missing those packages. `install/wsl-setup.sh` handles the `apt install` up front.

Fallback: if WSL playwright proves flaky, run the playwright MCP server on Windows natively and reach it from WSL over TCP (MCP over HTTP transport). Deferred until it actually bites.

### 3. Obsidian REST plugin must be running

The obsidian-vault MCP server proxies to `http://localhost:27124`, which is the Local REST API plugin inside Obsidian. Obsidian must be open on Windows with the plugin enabled. Not a code problem — a "Obsidian must be running" runtime dependency, same as it is today under Claude.

### 4. Bilingual output slippage

Per [[feedback-no-japanese-translation]], never translate JP→EN. Qwen2.5-Instruct is strong on JP but occasionally translates when asked to summarize a mixed-language page. Mitigation: system prompt calls this out explicitly; skills that handle bilingual content re-assert the rule in `skill_wrapper.md`.

### 5. Full skill parity is a *lot* of prompt-engineering iteration

Setting expectations on paper: v1 = "the harness runs, MCP works, ~10 read-heavy skills are reliable." Full parity across all ~30 skills is a multi-month iteration project, not a v1 deliverable. Don't measure v1 against Claude.

## Non-goals (out of scope)

- **Coding workflows** — that's [[agent-harness — Design Doc]]. This harness deliberately uses the *instruct* model, not the *coder* model. If you want to edit code with this, don't; use aider from `agent-harness/`.
- **Image understanding** — Qwen2.5-Instruct is text-only. Skip vision.
- **PDF ingestion beyond text extraction** — if a PDF needs OCR or layout parsing, the fetch/read tool returns raw text; complex parsing lives in a skill that shells out to `pdfplumber`, not in the model.
- **Persistent memory across sessions** — no auto-memory system in v1. Add if it turns out to matter after a month of use.
- **Fine-tuning Qwen** — off the table for v1. Prompt engineering only.
- **Multi-model routing** — one model per session. No "route hard prompts to a bigger model" logic.

## V1 → V2 progression

**V1 (build target):**
- Harness spawns 4 MCP servers, tool-call loop works end-to-end
- All skills discovered from vault, registered as slash commands
- REPL and one-shot modes both functional
- `qwen: skip` frontmatter honored
- Full skill parity *attempted* — expect ~5-10 reliable, rest need iteration

**V2 (as friction accumulates):**
- Per-skill prompt tuning (probably a `skill_prompts/<name>.md` override folder alongside the vault skill)
- Session persistence (`--resume <id>`)
- Sub-agent invocation from within skills (mirroring Claude's Agent tool)
- Windows-native Playwright fallback if WSL playwright stays flaky
- Autocomplete-model integration (qwen2.5-coder:1.5b) for `--complete` mode alongside chat — turns the harness into a general local LLM front-end, not just chat

## Open questions to resolve during build

- Does the obsidian-vault MCP server tolerate concurrent requests from parallel sub-agents, or does it serialize through the REST plugin? (Determines whether sub-agents can safely read/write vault at the same time.)
- Should `.claude/commands/*.md` be read from the vault or from a harness-local `commands/` folder? Vault means single source with Claude, but Claude and harness may want different command wrappers per skill. **Tentative: vault first, harness override folder if it becomes needed.**
- How does the harness handle a slash command that Qwen doesn't recognize as a valid tool call? (Prompt discipline vs. deterministic pre-processing — probably pre-process, i.e. detect `/foo` at the input layer and inject the skill content directly rather than hoping Qwen chooses the right tool.)
